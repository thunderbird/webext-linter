User-data exfiltration error

data-exfiltration

What it detects

Overt transmission of data to a remote host — through fetch, XMLHttpRequest, WebSocket, EventSource or navigator.sendBeacon. Sending user data such as message content, attachments, contacts or credentials off the machine is allowed only if the user actively enabled it, and Thunderbird has no built-in data-collection consent prompt.

A deterministic scan collects the overt outbound sinks whose destination is a remote or dynamically-built host. Whether a valid opt-in exists — a documented options-page setting where the user enters the destination, or an explicit off-by-default opt-in for a hard-coded URL — is a judgement call, so each such sink is handed to the model. Its evidence is the transmitting file plus the add-on's options page, where any opt-in would live. The model judges whether user data is transmitted without consent. Disguising transmission as a resource load is a separate, always-error concern handled by the disguised-* checks; this check covers only the overt channels.

When it fires The add-on is told the listed code locations send user data to a remote server without an explicit opt-in, and that user data may be transmitted only when the user actively enables it on an options page that discloses what is sent.
