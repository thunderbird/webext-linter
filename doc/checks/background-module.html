Background script needs type module error

background-module

What it detects

A background script that uses ES module syntax — static import or export — only loads when the manifest declares the background as a module. Without that declaration Thunderbird loads the script as a classic script and the module syntax fails.

The check looks at the scripts the manifest lists under the background entry (the scripts array and any service_worker). A classic background script with no module syntax is fine. A background page declared as HTML is handled by the sibling background-page-module check; dynamic import() is a call, not module syntax, and does not count.

When it fires The add-on is told the flagged background code uses ES module syntax while the background is not declared "type": "module", so Thunderbird will fail to load it, and must add "type": "module" to the background entry.

Source of truth: src/checks/rules/background-module.js and its entry in assets/registry.yaml.
