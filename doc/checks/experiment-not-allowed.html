Experiment APIs not accepted error

experiment-not-allowed

What it detects

An Experiment add-on whose experiment is not a recognised published Thunderbird API draft — an unknown API name, including one that shadows a built-in. Submissions like this are currently rejected, and the finding short-circuits the rest of the review.

This is the only check that runs in the invalid-experiment phase: when the pipeline decides an experiment is unsupported, it runs this check alone and nothing else. Recognised but locally modified drafts do not land here — they stay on the normal review path and are flagged by the experiment-modified check instead. The check stays silent when experiments are explicitly allowed and when the add-on is not an Experiment. For each unsupported experiment the reason is refined to "shadows a built-in" when its declared path actually resolves to a real API.

When it fires The developer is told that new submissions using Experiment APIs are not currently being accepted, except unmodified copies of the latest published API drafts, with a link to the upstream drafts and to the policy announcement.

Source of truth: src/checks/rules/experiment-not-allowed.js and its entry in assets/registry.yaml.
