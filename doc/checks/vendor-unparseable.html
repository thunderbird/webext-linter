Unparseable VENDOR file error

vendor-unparseable

What it detects

Developers declare every bundled third-party library in a VENDOR file so each can be verified against its upstream source. An entry is a bundled library file paired with a source URL that points to a file, both within one block (separated by headings or blank lines).

This check fires when a VENDOR file is present but nothing could be parsed out of it — no block pairs a recognised library with a file source URL. The add-on's own modules and the package name are not entries, and a bare repository link (such as a github.com/owner/repo page) is not a source. When the file declares a library that is merely missing from the package, that is the missing vendored file check instead, not this one.

When it fires The add-on is told that a VENDOR file is present but no third-party library could be parsed from it, and is asked to list each bundled library in its own block as the packaged file path plus a source URL that points to the exact file (on unpkg, jsDelivr, the npm registry, or a raw GitHub URL), so it can be verified automatically.

Source of truth: src/checks/rules/vendor-unparseable.js (the VENDOR parse lives in src/normalize/vendor.js) and its entry in assets/registry.yaml.
