Experiment code review manual

experiment-manual-review

What it detects

Every Experiment submission. Experiments run with full access to Thunderbird's internals, so beyond the automated checks they always need a careful human code review of the whole add-on.

This check carries no automated severity. When the add-on is an Experiment it emits a single whole-add-on escalation with no specific location, which surfaces as a manual-review reminder. It is silent for non-Experiments, and an unsupported experiment never reaches it because such a submission aborts earlier in the invalid-experiment phase via the experiment-not-allowed check.

Decision flow

flowchart TD
  A["A submission"] --> E{"Did the manifest parse and
is it an Experiment?"}
  E -- no --> S([skip])
  E -- yes --> MAN([escalate to manual review])

  classDef skip fill:#eef0f3,stroke:#9aa3ad,color:#5b626b;
  classDef manual fill:#eceef1,stroke:#5b626b,color:#3a3f45;
  class S skip;
  class MAN manual;
When it fires The reviewer is reminded that this is an Experiment submission requiring careful code review because it has full access to Thunderbird's internals, that the Experiment surface should be kept to a minimum and must not harm usability, stability, performance, or security, and that reviewing only the differences from the last approved submission can save time.

Source of truth: src/checks/rules/experiment-manual-review.js and its entry in assets/registry.yaml.
