Disguised data transmission error

disguised-transmission

What it detects

A covert channel — a resource src/href, a CSS url(), a window.open() or a page navigation — whose URL is built with a runtime value and points at a remote host, but that carries no obvious user-data API call. Slipping data out through a channel not meant for transmission is disguised exfiltration.

On its own, "a remote URL with a dynamic piece" is a weak signal — it is extremely common in legitimate code (navigating to a per-item page, loading an image whose name varies). The strong case, where a message/contacts-style data call sits directly in the URL, is caught as a hard error by the deterministic disguised-resource / -stylesheet / -window / -navigation checks. This check takes the weaker residue those leave behind and asks the model, one suspected location at a time, whether it is really an undisclosed data send. Each verdict maps directly: a send is an error, a legitimate dynamic URL is dropped, and anything the model cannot tell goes to manual review.

When it fires The add-on is told the listed code locations appear to send data through a resource or navigation URL — a channel not meant for transmission — that disguising a send this way is not allowed even with consent, and that it should use a normal request and disclose what is sent. Locations the model cannot resolve are deferred to the post-summary recheck.
