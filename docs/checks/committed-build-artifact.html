Committed build artifact error

committed-build-artifact

What it detects

A source-code submission that ships a committed binary archive — a .xpi, .zip, .crx, or other archive anywhere in the submission. A built archive is build output: the reviewer reproduces it from the readable source during the build, so a committed one is useless bloat and, worse, a decoy — its bytes can differ from what the build actually produces, presenting a clean-looking source next to a tampered artifact.

Archive paths are recorded when the submission is loaded, before it is split into the review source and the build files, so an archive is caught wherever it sits — in the review source, the build tree, anywhere in the submission. Each recorded archive becomes one error finding. This mirrors the committed node_modules reject.

Decision flow

When it fires The add-on is told that a committed build artifact is not allowed and must be removed before resubmitting: a built archive is produced from the source during the build, so it must never be part of the submission, where its bytes cannot be verified against what the build produces.

Source of truth: src/checks/rules/committed-build-artifact.js (fed by the archive recording in src/addon/load.js, using ARCHIVE_EXTENSIONS from src/util/files.js) and its entry in assets/registry.yaml.
