Unused, hidden or duplicate files or folders error

unused-files

What it detects

Files that should not ship in a published add-on: OS or editor junk and hidden dotfiles, and files that no part of the add-on ever loads. Such files complicate review, may leak information about the build system, and needlessly inflate the package.

A deterministic pre-flight follows every static load edge — imports, getURL, HTML and CSS references, and file-loading APIs — to decide what is reachable. Dependency manifests, the manifest, locale catalogs and documentation metadata are always exempt. Clear junk by name and a file named only by dead code with no dynamic loader are reported directly. When a file's name appears in live code, or the add-on builds load paths at runtime, the linter cannot tell statically whether any site really loads it, so each suspected loader site is handed to the model; the file is unused only if no site loads it.

Decision flow

does this site load the file?"}
  L -- "at least one site loads it" --> OK
  L -- "no site loads it" --> ERR2
  L -- "unsure / no verdict" --> M([escalate to manual review])

When it fires The add-on is told the listed files or folders appear unused or hidden, that they complicate review and bloat the package, and is asked to fix its packaging so they are no longer included.

Source of truth: src/checks/rules/unused-files.js and its entry in assets/registry.yaml.
