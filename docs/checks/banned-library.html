banned-library

What it detects

A bundled third-party library pinned to a version that Mozilla's add-on policy disallows (banned) or discourages (unadvised) — for example jQuery before 3.0, or AngularJS before 1.5.9 (a Content-Security-Policy bypass). This is a policy check, distinct from Vulnerable dependency: it fires on a version Mozilla bans even when no published CVE advisory exists, and it covers a package.json dependency, an npm-sourced VENDOR entry, and a library the linter identified by content hash.

The policy is a small curated list — assets/library-blocks.yaml (overridable with --lib-mozilla-block-db) — because Mozilla publishes no machine-readable block database; it is kept in sync by hand from the addons-linter documentation. Before the review, every pinned or identified shipped library is matched against this policy, once, in the same network pre-step as the OSV audit. A hit is recorded with its version and the policy reason; a banned hit also skips the OSV query for that library (it is rejected regardless). This check reads that result and emits one finding per hit, anchored at the declaration line (or, for an identified library, at the bundled file). Its severity is auto: banned is an error, unadvised a warning. Build-only devDependencies are never shipped, so they are not policy-blocked here.

When it fires The add-on is told a bundled library version is disallowed (or discouraged) by Mozilla's add-on policy, with the version and the reason, and is asked to update to a permitted version and re-bundle.

Source of truth: src/checks/rules/banned-library.js, src/checks/lib/library-blocks.js, the curated assets/library-blocks.yaml, and its entry in assets/registry.yaml.
