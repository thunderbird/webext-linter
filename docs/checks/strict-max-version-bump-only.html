strict_max_version can be updated on ATN info

strict-max-version-bump-only

What it detects

A resubmission whose only meaningful change is a bump to the version and the gecko strict_max_version. When nothing else in the add-on has changed, the developer could have raised the maximum supported Thunderbird version directly on the add-on's ATN admin page instead of uploading a whole new version.

This is a diff check: it runs only when a baseline to compare against is supplied. It compares file contents rather than the archive, so a re-zip with different ordering or timestamps is not mistaken for a real change. Every non-manifest file must be byte-identical, and the manifest must differ only in the version and strict_max_version once key order is normalised.

When it fires The developer is told that instead of submitting a new version that only raises strict_max_version, they can update the maximum version of the most recent submission directly on the add-on's ATN admin page.

Source of truth: src/checks/rules/strict-max-version-bump-only.js and its entry in assets/registry.yaml.
