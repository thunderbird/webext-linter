Vendored library not vulnerability-checked info

vendor-vuln-unknown

What it detects

A vendored library — typically sourced from a GitHub or CDN URL — that could not be matched to a known npm package, so its known-vulnerability status could not be determined automatically.

Before the review, every GitHub-sourced VENDOR entry is run through a resolution step that tries to prove an npm identity for the bundled bytes by content-hash match: first the repository name at the declared ref, then, if the model is enabled, an LLM-proposed name re-proved by the same hash. Entries it can resolve are audited against the OSV database by the vulnerable-dependency check instead. First-party trusted-org GitHub sources are accepted by provenance and produce nothing here. Whatever remains unresolved is recorded as unaudited, and this check surfaces one informational note per entry so the reviewer knows it went unchecked.

When it fires The add-on is told a third-party file was sourced from a CDN where its known-vulnerability status could not be determined automatically, and is nudged to switch to an npm-hosted source (a unpkg or jsDelivr package pinned to an exact version) so it can be audited against the OSV database.

Source of truth: src/checks/rules/vendor-vuln-unknown.js and its entry in assets/registry.yaml.
