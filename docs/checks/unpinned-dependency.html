Unpinned dependency error

unpinned-dependency

What it detects

A dependency declared in package.json as a version range with nothing to pin it to an exact release. A range can resolve to different versions over time, so the bundled copy cannot be verified against a known upstream.

Before the review begins, the vendor resolver reads the package.json dependencies once. A dependency given as an exact version is considered pinned; a range is accepted only if a committed lock file (package-lock.json, npm-shrinkwrap.json, yarn.lock, or pnpm-lock.yaml) records the exact version used. Specs that point at git, a URL, a file, or a workspace are not npm-verifiable and are skipped. Whatever remains unpinned is recorded, and this check turns each one into a finding.

When it fires The add-on is told a bundled dependency is not pinned to an exact version, that a range can resolve to different releases over time, and that it should pin the version in package.json or commit a lock file recording the exact version used.

Source of truth: src/checks/rules/unpinned-dependency.js and its entry in assets/registry.yaml.
