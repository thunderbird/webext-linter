Possibly unused permissions warning

unused-permission

What it detects

This check is a producer: it enumerates every declared named permission that a reachable call does not provably require. A permission a reachable call provably needs is dropped — whether a WebExtension API call requires it, or (for permissions the schema gates on a Web/DOM API instead of a browser.* member, like clipboard and geolocation) a matching navigator.* call proves it used. A permission whose registry prompt entries declare usage tokens (the code-level spellings of its justifying usages, e.g. compose_scripts, executeScript, insertCSS for compose) that appear nowhere in the add-on's live code (comments excluded) or manifest is deterministically unused — a warning finding, with or without --llm-review. The deterministic path stands down whenever the scan is blind: an SCA review (build-time dependencies are invisible in the source corpus) or unresolved dynamic API surface (computed member chains, unresolvable aliases). Every remaining permission escalates for human review. Host match patterns are out of scope here — they belong to the minimize-host-permissions check.

Under --llm-review, the escalated permissions the registry has a recheck prompt for are handed to the unused-permission-recheck consumer to be re-judged with full context; every other escalated permission stays a manual-review reminder. Without --llm-review, all of them stand as the reminder. This one producer serves every add-on: the version-specific tabs wording and token lists (the D308076 fix in Thunderbird 154) are selected from the add-on's strict_min_version.

Decision flow

flowchart TD
  A["The add-on manifest"] --> P["Each declared named permission
(required + optional)"]
  P --> H{"Is it a host match pattern?"}
  H -- yes --> OK([no finding — host check's concern])
  H -- no --> U{"Provably required by a

  U -- yes --> OK2([no finding — justified])
  U -- no --> T{"Prompt entry declares usage tokens,
and none occur in the live code

  T -- yes --> WARN(["warning finding — deterministically
unused, the model is never asked"])
  T -- no --> G{"Does the registry have a recheck

  G -- yes --> ESC(["handed to the unused-permission-recheck

else a manual-review reminder"])
  G -- no --> MAN(["manual-review reminder only
— never sent to the model"])

When it fires The add-on is told that one or more declared permissions appear to be unused and is asked to remove any it does not need, or to explain which part of the add-on stops working without them.

Source of truth: src/checks/rules/unused-permission.js, enumerateUnusedPermissions in src/checks/lib/permissions.js, and its entry plus the per-permission tokens in assets/registry.yaml.
