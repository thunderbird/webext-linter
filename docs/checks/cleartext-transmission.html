Unencrypted data transmission error

cleartext-transmission

What it detects

Data sent to a remote host over an unencrypted connection — a URL using one of the non-TLS schemes http://, ws:// or ftp://. Anything sent in cleartext can be read or tampered with in transit, so this is an error whatever the payload is; there is no test for what is actually sent.

The check looks only at the developer's own JavaScript and only at overt transmission APIs — fetch and its relatives such as XMLHttpRequest, WebSocket, EventSource and sendBeacon — aimed at a remote host. Covert loads that smuggle data through a URL are already a hard error in the disguised-* checks, so they are deliberately not reported here to avoid double-reporting the same line. An encrypted remote transmission over https://, wss:// or ftps:// is fine.

When it fires The add-on is told that data is sent over an unencrypted connection to the named host, and that all data must travel over an encrypted channel — use https:// (or wss:// for sockets), never http://, ws:// or ftp://.

Source of truth: src/checks/rules/cleartext-transmission.js and its entry in assets/registry.yaml.
