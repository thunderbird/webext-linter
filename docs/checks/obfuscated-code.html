Obfuscated code is not permitted error

obfuscated-code

What it detects

This check flags the developer's own JavaScript when it ships obfuscated — code deliberately written to hide its behavior — and so cannot be reviewed as it stands. Obfuscation is recognized structurally: the file is matched against the AST shape of a known obfuscator family (obfuscator.io, array- and function-replacement transforms, caesar-plus) via the obfuscation-detector library.

Detection is two-tier. A strong family match is a deterministic finding. One family — the revealing-module pattern (const X = (() => {…})(); X.init();) — also fits ordinary readable code, so a weak-only match is not decided outright: the file becomes a single candidate the model judges from its own content, mapping fail → finding, unsure → manual review, pass → drop (with no model configured, every candidate falls to manual review). The model is never told which family the detector matched, so its judgment cannot be anchored to a detector claim.

A file that looks like a recognised third-party library is excluded here, because that is the missing-library check's concern, and a file that is merely minified (not obfuscated) is the minified-code check's concern. The detection has high precision but partial recall, so it cannot catch every obfuscator.

When it fires The add-on is told the named files are obfuscated and that obfuscated code is not permitted — whether or not the add-on is hosted on addons.thunderbird.net — as it violates the Add-on Policy and is subject to being blocked. It is asked to ship the original, human-readable code directly. A weak-family match the model cannot decide (or that no model was configured to judge) is deferred to manual review instead.

Source of truth: src/checks/rules/obfuscated-code.js and its entry in assets/registry.yaml.
