Unreviewable build dependency error

unpopular-source-dependency

What it detects

A declared dependency that could not be confirmed to be a widely-used library. This check runs only for a source-code submission, where the dependency code is pulled in at build time: it is absent from the readable source and mangled in the built add-on, so a dependency that is not trusted by ubiquity cannot be reviewed at all.

Before the review begins, a network pre-step looks up the popularity of each package.json dependency: monthly downloads for an npm package, or stargazers for a GitHub source. A dependency below the trust bar is recorded, and this check turns each one into a finding anchored at its declaration line. The lookup is deliberate: a failed or offline lookup records nothing, so the reject only ever fires on a confirmed reading below the bar. A dependency that clears the bar is trusted by ubiquity and never reaches here.

Decision flow

flowchart TD
  A["A package.json dependency"] --> G{"Source-code submission?"}
  G -- no --> S([skip])
  G -- yes --> B{"Popularity lookup succeeded?"}
  B -- "no (offline or failed)" --> S
  B -- yes --> C{"Above the widely-used bar
(npm downloads or GitHub stars)?"}
  C -- yes --> OK([no finding])
  C -- no --> ERR([ERROR: unreviewable build dependency])

When it fires The add-on is told a declared dependency could not be confirmed to be widely-used, and that it needs to be packaged with the add-on so it can be reviewed.

Source of truth: src/checks/rules/unpopular-source-dependency.js and its entry in assets/registry.yaml.
