API newer than strict_min_version error

strict-min-version-api

What it detects

The add-on declares a lowest supported Thunderbird release through strict_min_version, but calls a WebExtension function or event that was only added in a later release. On those older installs the API does not yet exist, so an unconditional call throws.

Only real, schema-resolved APIs are considered — a misspelled or non-existent name is left to the unknown-API check, so feature detection cannot launder it. The same too-new API can be used safely if it is guarded so it runs only where it exists (optional chaining, a typeof or existence test, or a runtime version gate). A call that carries such a guard signal is the ambiguous case: only there does the linter ask the model whether the guard truly protects the call. An API used unguarded anywhere becomes the hard finding outright.

When it fires The add-on is told it sets strict_min_version too low while using functions or events added in a later Thunderbird, and is asked to raise strict_min_version or guard the call.

Source of truth: src/checks/rules/strict-min-version-api.js and its entry in assets/registry.yaml.
