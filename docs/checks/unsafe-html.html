Modifying innerHTML, unsanitized HTML warning

unsafe-html

What it detects

Dynamic content written into an HTML sink — innerHTML, outerHTML, srcdoc or insertAdjacentHTML. If that content can carry external or user-provided data and is not sanitized, it is an injection risk.

This is a warning rather than an error: the construct is only dangerous when the content is untrusted and unsanitized, which a static scan cannot fully determine, so the check points the reviewer at each site instead of deciding for them. Plain static-string content is not flagged. Bundled, minified or otherwise non-authored files are skipped. Each remaining dynamic-content sink hit becomes a finding naming the sink involved.

When it fires The add-on is told that dynamic content is assigned via the named sink, and that if it can contain external or user-provided data it should be sanitized (with DOMPurify or the Sanitizer API) or built using textContent and createElement, with a link to the guide.

Source of truth: src/checks/rules/unsafe-html.js and its entry in assets/registry.yaml.
