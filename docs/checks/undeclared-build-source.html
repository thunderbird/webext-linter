Build process review error

undeclared-build-source

What it detects

In a source-code submission the reviewer builds the add-on from the readable source, but the dependency audit only reads the root manifest and lock file. A build script or config could ignore the manifest and pull code or resources from an undeclared source, leaving the declared dependencies a clean decoy.

The build is classified once, up front, by the build analysis: a single model request in the setup phase over the build-relevant files (collected by following the package manifest, so build output and tooling the build never runs are never sent). This check reads that stored classification. A remote-fetch classification — the build pulls code or a resource from an undeclared external source — is the error. It also owns the fallback lane: when the build could not be classified with confidence (offline / no token, or a step the linter could not statically follow, such as an opaque orchestrator or a network fetch) the whole build routes to extended manual review. The other build classification (not-from-source) is owned by its own check; a genuine build, and a submission with no manifest to follow, produce nothing here.

When it fires On a substantiated failure the add-on is told, with the model's explanation, that every part of the build must come only from the dependencies declared in the manifest and lock file. Otherwise the build is escalated to extended manual review, where a reviewer reproduces the build by hand — using the build steps the model recovered — and confirms it pulls only from the declared dependencies.

Source of truth: src/checks/rules/undeclared-build-source.js (fed by the setup build analysis, src/build/analyze.js) and its entry in assets/registry.yaml.
