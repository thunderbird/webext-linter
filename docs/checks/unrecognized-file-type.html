Unrecognized referenced file type error

unrecognized-file-type

What it detects

A packaged file the add-on loads — declared in the manifest, or pulled in by a <script src> in a live page — whose file type the review tools do not recognize. The browser still loads it, but no check could classify or review it, so it went unreviewed.

The set of files the tool parses as code is a suffix guess (JS_EXTENSIONS). Where that guess is narrower than what the browser actually executes — a script shipped under an unusual extension — the file would silently escape every code check. This check is the backstop: instead of trying to enumerate every possible code extension, it flags anything the add-on references whose suffix is in no recognized type at all. An un-enumerated suffix stops being a silent gap and becomes a loud finding a reviewer must resolve.

Detection happens where reachability already resolves references: the manifest walk (role-blind — any declared file with an unrecognized suffix) and the <script> walk (a script target with an unrecognized suffix). Only files that are actually reachable are reported; a reference in dead code never runs. The check reads the built XPI (input: xpi), so in a source-code submission it can surface a file the developer’s packaging step emitted.

When it fires The add-on is told which packaged file has an unrecognized type and where it is referenced, and asked to give it a standard extension for its content (for example a .js script), or to confirm what the file is and why it ships. If it is a build artifact, the developer is asked to review what their packaging step produces.

Source of truth: src/checks/rules/unrecognized-file-type.js (reading RECOGNIZED_EXTS from src/util/files.js and unrecognizedRefs from src/lib/reachability.js) and its entry in assets/registry.yaml.
