Missing third-party library reference info

missing-library

What it detects

This check reports a bundled JavaScript or CSS file whose content hash matches a known third-party library release but that is not declared in a VENDOR file. The hashes come from a known-library database (Mozilla dispensary's hashes.txt), fetched and cached like the schema. Because it is a byte-exact match, the check names the exact library and version it found — for example bootstrap 5.3.0.

It is the backup for a forgotten VENDOR declaration: declared and experiment-trusted files are excluded before the lookup, so everything it reports is genuinely undeclared. Since a match is a precise, byte-verified release, the finding is advisory (info), not a blocker. A bundled file the database does not recognise is not treated as a library here — if minified it is reported by minified-code, if obfuscated by obfuscated-code, and otherwise it is scanned as the developer's own source.

Because the match identifies an exact release, an identified library is also audited for known vulnerabilities just like a declared dependency — an undeclared but vulnerable bundle is reported by vendor-vulnerable (and stays excluded from --scan-minified, since it is real third-party code, not the developer's).

When it fires The add-on is told which third-party library and version each file was recognised as, and is asked to declare it and link its source — commonly via a VENDOR file at the add-on root — so the bundled copy can be verified against the linked release.

Source of truth: src/checks/rules/missing-library.js and its entry in assets/registry.yaml.
