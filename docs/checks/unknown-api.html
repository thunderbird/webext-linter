Unknown or unsupported WebExtension API error

unknown-api

What it detects

Calls to WebExtension APIs (browser.*, messenger.*, chrome.*) that Thunderbird's annotated schema does not provide: an unknown namespace, an unknown member of a known namespace, or a member that exists but is explicitly marked unsupported.

Only the plain WebExtension code is checked — Experiment and privileged code legitimately use Thunderbird internals rather than the WebExtension schema, and dead code never runs, so both are excluded.

A namespace that the add-on's own Experiments define (the base namespace of any experiment_apis entry, for example calendar) is treated as known: the developer owns that API, so the whole namespace and everything under it is accepted rather than tracing each sub-API.

A reference that is feature-detected — guarded by a typeof/existence check or optional chaining, including through a captured namespace alias — is skipped rather than flagged when it is an unknown member or an unsupported API: the add-on's fallback runs where the API is missing, so nothing breaks. An unknown namespace is still flagged even when guarded — a whole missing namespace is far more likely a typo or hallucination than a compatibility probe.

When it fires For an unguarded unknown/unsupported API (or a guarded unknown namespace), the add-on is told the named API is not supported in the targeted Thunderbird version, with a link to the WebExtension API documentation. A guarded unknown member or unsupported API is skipped, not reported.

Source of truth: src/checks/rules/unknown-api.js and its entry in assets/registry.yaml.
