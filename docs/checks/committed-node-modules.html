Committed node_modules folder error

committed-node-modules

What it detects

A source-code submission that ships a committed node_modules folder. Installed dependencies are build output — the reviewer installs them from the declared package.json and lock file at build time — so a node_modules folder anywhere in the source root is useless bloat and, worse, a decoy: the committed bytes can differ from the declared dependencies, smuggling in code from an undeclared source.

node_modules is never read. When the submission is loaded, a node_modules directory is skipped and never opened; only its path is recorded, so a huge committed tree costs nothing and its contents can never influence the review. This holds anywhere in the source root, including inside the add-on source folder. Each recorded node_modules directory becomes one error finding.

When it fires The add-on is told that a committed node_modules folder is not allowed and must be removed before resubmitting: the declared dependencies are installed from package.json and the lock file during the build, so node_modules must never be part of the submission.

Source of truth: src/checks/rules/committed-node-modules.js (fed by the node_modules skip in src/addon/load.js) and its entry in assets/registry.yaml.
