Source archive not required warning

sca-not-required

What it detects

A source-code archive (--sca-root) that was not needed, because the shipped XPI is already directly reviewable — its first-party code is not minified or obfuscated. SCA mode exists only to review the readable source of an add-on whose built XPI cannot be read; when the XPI itself is readable, the separate source submission adds nothing.

The decision is made purely from the built XPI's own classification (the same minified/obfuscated first-party signal minified-code and obfuscated-code use, recognized libraries excluded) — the source archive's content is never consulted. If the XPI is directly reviewable the pipeline downgrades to a plain XPI review (so minified-code and the rest still run against the XPI) and this check raises one warning, anchored at the manifest. If the shipped XPI is minified/obfuscated the source archive is kept and this never fires.

Decision flow

When it fires The add-on's built XPI is directly reviewable, so a separate source-code archive is not required; it was reviewed as a standard XPI. A source-code archive switches the add-on into a separate, more involved review process that is considerably slower — since the shipped code is already human-readable, submitting only the XPI as a standard review avoids that delay. A source-code submission is needed only when the shipped code is minified or otherwise not human-readable.

Source of truth: src/checks/rules/sca-not-required.js (fed by the downgrade decision in src/pipeline.js and hasUnreviewableCode in src/lib/bundled.js) and its entry in assets/registry.yaml.
