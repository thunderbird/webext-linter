Vulnerable dependency auto

vendor-vulnerable

What it detects

A third-party library whose bundled version has published security advisories. This covers a package.json dependency, an npm-sourced VENDOR entry, and — even when it was never declared — a library the linter identified by content hash (so a developer who bundled a vulnerable release without declaring it is still caught).

Before the review, every pinned npm package and version — plus each hash-identified library — is audited once against the OSV vulnerability database over the network. Each package with known advisories is recorded with the advisory ids, the affected version, the fixed versions, and a severity band. This check reads that result and emits one finding per vulnerable package, anchored at its declaration line (or, for an identified library, at the bundled file). Its severity is auto: rather than a fixed level, each finding takes its severity from the advisory band — high or critical becomes an error, moderate or medium becomes a warning, and anything else (low or unknown) is informational.

When it fires The add-on is told a bundled dependency has known security vulnerabilities, with the affected version, the advisory ids, and the severity, and is asked to update to a fixed release, re-bundle, and resubmit — or to explain why the advisory does not apply to how the add-on uses the dependency.

Source of truth: src/checks/rules/vendor-vulnerable.js and its entry in assets/registry.yaml.
