Source Code Archive review flow

verify.js <xpi> --sca-root <src> [--sca-source <dir>]

What it reviews

Some add-ons are submitted as both a built XPI (minified, what users install) and a readable source archive. Reviewing the minified XPI directly is noisy, so SCA mode reviews the readable source for code defects while still treating the built XPI as the authoritative shipped artifact.

SCA mode is on when --sca-root (the source archive holding package.json / the lock file) is set. --sca-source (the add-on code root within it) is optional and defaults to . - the whole root reviewed as the source, for a flat layout with manifest.json at the root. The review then splits across five inputs with a fixed role each, and each check is routed to exactly the artifact its input selects. The purple nodes are the optional LLM side-steps (only with --llm-review); Experiment handling and the summaries work as in the Standard flow.

Where the LLM is used

The purple steps are optional and gated on --llm-review, sharing the same run-wide request cap as a Standard review. In SCA mode the LLM is consulted at:

  • Pre-flight — validate the provider and model, hard-fail (exit 2) if unusable.
  • Vendor resolution — parse ambiguous package.json declarations from the source archive.
  • The source LLM checks — data-exfiltration, remote-script, remote-eval, unused-files, minimize-web-accessible-resources, missing-english-localization — escalate their ambiguous residue over the readable source.
  • The build-process review — one setup model call (analyzeBuild) classifies the build over the files reached from package.json, and three deterministic input: build checks gate on the result: undeclared-build-source (the build fetches code or a resource from an undeclared source — a raw URL, curl | sh, an unpinned git clone, a CDN, a postinstall hook — plus the offline/unresolved manual-review fallback), build-not-from-source (the XPI is packaged from committed artifacts, not built from the source), and build-source-redundant (the build only vendors node_modules, so a source submission is not needed).
  • The summaries + recheck — --llm-review runs two passes in SCA: a behavioral summary over the readable source (the human-facing "Summary of add-on", re-judging the source-anchored rechecks — data exfiltration, disguised transmission, unused permissions) and a recheck-only packaging summary over the built XPI (the XPI-anchored rechecks — unused files, web_accessible_resources, localization). Each pass re-judges only its corpus's escalated unsure items. With --diff-to, a summary of changes describes the built XPI.
One review, five inputs with fixed roles The readable source is the deterministic review target and the subject of the behavioral summary; the source package.json supplies the audited dependencies; the build files are reviewed on their own; the built XPI is authoritative for the experiments, the file-completeness checks, the diff baseline and the packaging summary; and the pure-manifest checks read the shipped manifest on a context with no file corpus at all (input: manifest). A check derives one from the other only through its declared input, so the source-level checks never touch the shipped bytes and the manifest checks never touch any file. The Standard flow is the same machinery with those roles collapsed onto a single add-on.

Because a review spans two submitted artifacts, every finding's file:line is prefixed with the artifact it lives in — [XPI] for a file in the built XPI, [SCA] for a file in the readable source code archive — and the Issues section closes with a legend explaining the two. The same prefix appears on the live activity feed. A Standard (XPI-only) review has one artifact, so it adds no prefix.
