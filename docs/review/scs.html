Source Code Submission review flow

verify.js <xpi> --scs-root <src> --scs-source <dir>

What it reviews

Some add-ons are submitted as both a built XPI (minified, what users install) and a readable source archive. Reviewing the minified XPI directly is noisy, so SCS mode reviews the readable source for code defects while still treating the built XPI as the authoritative shipped artifact.

SCS mode is on when both --scs-root (the source archive holding package.json / the lock file) and --scs-source (the add-on code root within it) are set. The review then splits across four inputs with a fixed role each, and each check is routed to exactly the artifact its input selects. The purple nodes are the optional LLM side-steps (only with --llm-enabled); Experiment handling and the summaries work as in the Standard flow.

Where the LLM is used

The purple steps are optional and gated on --llm-enabled, sharing the same run-wide request cap as a Standard review. In SCS mode the LLM is consulted at:

  • Pre-flight — validate the provider and model, hard-fail (exit 2) if unusable.
  • Vendor resolution — parse ambiguous package.json declarations from the source archive.
  • The source LLM checks — data-exfiltration, remote-script, remote-eval, unused-files, minimize-web-accessible-resources, missing-english-localization — escalate their ambiguous residue over the readable source.
  • The build-process review — undeclared-build-source (input: build) has the model judge whether the build pulls code or resources from a source that is not among the declared dependencies (a raw URL, curl | sh, an unpinned git clone, a CDN, a postinstall hook).
  • The summaries + recheck — the behavioral --full-summary / --diff-summary describe the built XPI, and the full-summary pass re-judges the escalated unsure items with whole-add-on context.
One review, four inputs with fixed roles The readable source is the deterministic review target; the source package.json supplies the audited dependencies; the build files are reviewed on their own; and the built XPI is authoritative for the manifest, the experiments, the file-completeness checks, the diff baseline and the behavioral summary. A check derives one from the other only through its declared input, so the source-level checks never touch the shipped bytes and the manifest-coupled checks never touch the source. The Standard flow is the same machinery with those roles collapsed onto a single add-on.
