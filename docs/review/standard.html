Standard review flow

verify.js <xpi|folder>

What it reviews

The default mode: one submitted artifact — a built XPI archive or an unpacked add-on folder — reviewed directly against the matching Thunderbird schema and the ATN review policies. The same artifact is both the code under review and the shipped package.

The pipeline loads the add-on, picks the schema branch from its manifest_version and the chosen channel, classifies any bundled Experiment, resolves and verifies its vendored libraries, then runs the deterministic, LLM, and manual checks over it. The purple nodes below are the optional LLM side-steps — they only run with --llm-enabled; without it, each escalates to manual review instead. The tool is read-only: it never modifies or repacks the submission.

Where the LLM is used

Every purple step above is optional and gated on --llm-enabled with an LLM_API_KEY. All of them draw on one run-wide request cap (MAX_LLM_REQUESTS_PER_RUN): when it is hit the CLI prompts to run more, and otherwise the remaining LLM work escalates to manual review.

  • Pre-flight — when the LLM is enabled, a Setup step validates the provider config and that the model is reachable, and hard-fails the whole run (exit 2) if not.
  • Vendor resolution — the model parses ambiguous package.json / VENDOR declarations, and resolves a GitHub source to its npm package when the deterministic match is inconclusive.
  • The LLM checks — remote-script, remote-eval, data-exfiltration, unused-files, minimize-web-accessible-resources and missing-english-localization. Each always runs its deterministic pre-flight; only the genuinely-ambiguous residue is sent to the model, which returns fail / pass / unsure.
  • The summaries — --full-summary (an add-on summary + permission review) and --diff-summary (what changed since --diff-to). --llm-review is the shorthand for --llm-enabled --full-summary.
  • The post-summary recheck — the full-summary pass re-judges every unsure item other checks escalated, with whole-add-on context, so confident cases resolve instead of landing in manual review.

Experiment handling

Every add-on that declares experiment_apis is classified against the upstream published drafts first, and that classification — not the --allow-experiments flag — decides the path:

  • Unsupported + experiments off → rejected outright: the review short-circuits to the single experiment-not-allowed finding, with no vendor setup, no LLM, no summaries and no manual list.
  • Recognised, or modified, or accepted with --allow-experiments → the full review runs. The Experiment's API namespaces are registered into the schema so the developer's calls into them resolve, and the Experiment checks (experiment-modified, experiment-manual-review, experiment-overrides-api, experiment-missing-strict-max-version) run as part of the normal check loop.
The two artifacts collapse into one In a standard review the code under review and the shipped artifact are the same object, so every check — including the manifest-coupled file-completeness checks and the behavioral LLM audit — reads that one add-on. The Source Code Submission flow is the variant where those two roles split across a readable source archive and a separately-built XPI.
