The review pipeline

architecture · src/pipeline.js

A review is a single call to runPipeline() (src/pipeline.js), the tool's core: it performs an analysis of the provided add-on and returns a Review - the findings and manual-review notes. The call loads and resolves the add-on, builds the run context, then hands off to the orchestrator, which runs the checks. Everything below happens inside it.

Setup

Nothing is checked until the add-on is loaded, classified, resolved and parsed - the pipeline's first job, and it does every expensive thing exactly once. The amber boxes are the ones that matter: they resolve the review state - invalidExperiment, the mode (with scaNotRequired), llmVerified, and the review target - and everything after them, every check, keys off those. After Phase 4, Phase 5 runs the review (a single runChecks() call, detailed in The orchestrator) then finalizes the Review for the report.

The SCA → XPI downgrade. A source archive is only worth reviewing when the shipped XPI cannot be. So resolveReviewMode() classifies the built XPI and decides from its own code alone: if the first-party code is not minified or obfuscated, the source adds nothing. The review is downgraded to a plain XPI review, the source archive is never read, and sca-not-required reports it. An invalid Experiment skips the classification entirely, so it is never downgraded.

The invalid-Experiment path is not an early exit - it runs on to the orchestrator, which reviews the shipped XPI with the single reject check. It only skips work, in one step: setting invalidExperiment pins the mode and bypasses everything after it - the LLM (pre-flight and client, even under --llm-review), the library hashes, the classification, the vendor block, the parse. Not one line of the add-on's code is read, so ctx.jsSources is empty; the reject check judges from the manifest, the experiment classification and the schema - which is resolved before the decision because the check resolves API paths through it.

The stores

What setup leaves behind is a set of shared stores. The diagram reads left to right in four columns: the producer step, the store it builds, the access layer, and the checks. A store (solid) is raw and eager - built once during setup (ctx.schema, addon.vendor/bundled, the per-source src.extracted.* fields), whether or not anything reads it. A cache (dashed) is a derived, lazy view - apiResolution, permissionAnalysis, outboundSinks, evalScan, locales, and the reachability graph - computed from stores on the first read that needs it, then memoized (so a cache nobody reads is never built). The lazy caches never re-parse: they derive from the precomputed src.extracted stores (a check reading before the extraction pass ran throws "a check is a pure reader, it never parses"). The small unlabeled circles are relays - pure layout, so a store a check reads directly still lines up in the last column. Solid arrows are "produces / derives from"; dashed are "read by".

A check is a pure reader. It never parses, never fetches, never mutates. So two checks asking the same question always agree, and peak memory stays at a single AST, regardless of how much code the add-on ships.
store built by what it holds
addon.vendor resolveVendor() → verifyVendor() The declared third-party libraries, resolved to their sources and verified over the network, plus the vulnerability audit. The vendor checks are pure readers of it.
addon.bundled classifyFiles() → assembleBundled(), finalized by identifyBundledLibraries() The per-file classification (authored / library / minified / obfuscated / vendored) and the nonAuthored skip set that every content scanner honours. identifyBundledLibraries() (the CDN + not-popular-vendor reconciliation) finalizes the split - it both adds and removes - which is why the parse runs after it.
src.extracted
(per ctx.jsSources entry) 		runExtractionPass() Each source is parsed once; the small per-file results (API usage, loader refs, network sinks, remote-code refs, the comment-free code atoms …) are stored and the AST is dropped. This is why no check ever re-parses.
ctx.schema resolveReviewSchema() The SchemaIndex for the selected channel, with any Experiment namespaces registered into it, plus the local annotation overlay.
the lazy ctx caches memoized on first read permissionAnalysis, apiResolution, evalScan, outboundSinks, locales … - derived on demand by whichever check asks first from the precomputed stores (no re-parse), then shared with the rest.

Which stores a check may reach is not open either. Every check declares an input, and routeCtx - the single place routing is decided, shared by the main loop and the deferred loop so they can never drift - hands it the sibling ctx for that input. Routing is total: source is a first-class sibling like the rest, there is no default artifact to fall through to, and a declared input with no sibling (a stray input: build in an XPI review) throws rather than silently running on the wrong corpus. A check reads exactly one artifact and cannot reach another's. What the input resolves to depends on the review mode:

input SCA review (readable source + built XPI) XPI review (one artifact)
source the readable source (siblings.source) the built add-on (siblings.source)
xpi the built XPI (siblings.xpi) the built add-on (siblings.xpi)
build the SCA build files (siblings.build) — (SCA-only)
manifest the shipped manifest (no file corpus) the shipped manifest

In an XPI review there is one artifact, so siblings.source and siblings.xpi are the same ctx. The one input-less case is a post-summary recheck consumer: the loader forbids it an input, so routeCtx routes it to siblings.source - it needs the review-level ctx.recheck / ctx.recheckVerdicts that live there - even though the items it re-judges belong to its producer’s artifact. Each handed item already carries every locus it needs (the producer stamps them before diverting), so the consumer reads no other artifact. The producer corpus is still recovered by ctxForRule (via labelInput) for the report’s [XPI]/[SCA] labels and the folder-collapse.

The orchestrator

With the stores in place runChecks() takes over. A check's phase is simply the registry.yaml section it lives in - never declared on the entry, so the two cannot disagree. The orchestrator looks up the phases it wants, in the order it runs them, which makes any section it never asks for inert.

phase (= section) when it runs run by example check
invalid-experiment-phase ONLY when the add-on is an invalid Experiment (ctx.invalidExperiment) - and then it is the only phase that runs at all: the whole rest of the review is skipped. runChecks experiment-not-allowed
deterministic-phase First phase of a normal review. runChecks missing-permission
llm-phase Straight after the deterministic phase. runChecks strict-min-version-api
post-summary-phase Last, after the add-on-summary interleave has filled ctx.recheckVerdicts — still inside runChecks. runChecks unused-permission-recheck (+ 5 more)

runChecks() (src/checks/registry.js) runs the whole review: the deterministic and llm phases in its main loop, then the add-on-summary interleave (resolveRecheckSummaries() in src/checks/summaries.js) that fills ctx.recheckVerdicts, then the post-summary phase that reads them, then the unused-folder collapse, and finally the per-site recheck verdict rows the report prints. It returns the finished findings, manual items, the checks that ran, the summaries and those rows (recheckVerdictRows); runPipeline()’s Phase 5 (src/pipeline.js) just calls it and assembles the Review. Every check, in every phase, goes through the same per-check runner, runOneCheck(). The phase runner is runChecks()’ main loop (src/checks/registry.js): it walks a phase’s gated checks in order, routes each to its input’s ctx (routeCtx), runs it through runOneCheck(), and collects the results - findings onto the returned list, manual items either straight to manual review or, for a post-summary-recheck producer, diverted into ctx.recheck for the AI summary to re-judge. runOneCheck() decides what to do with a check purely from the shape its run() returned - never its phase; a check that throws becomes a single check-failed finding so the rest still run; and only the orchestrator routes anything to manual review. This loop is also where the lazy caches are computed - the first check to read one builds it inside its run() and memoizes it on ctx.addon for every later check (see The stores).

Interpreting the model's answer

There are two seams for deciding what the model's response means, and they are not symmetric.

  • An llm-phase check returns a resolve closure from its own run(), so it fully owns the interpretation of its output.
  • A post-summary consumer has no such hook. Five delegate to the shared resolveRecheck() - read the handed items and their verdicts, apply one uniform mapping (pass → drop, fail → finding, else manual). The unused-permission-recheck consumer instead uses resolvePermissionRecheck(), because its items are judged per token site and must be aggregated back up to the permission (see below).

These two seams are not separate worlds - they form a funnel. A direct llm-phase check judges its candidates against a focused corpus (just the candidate’s files, batched) and settles the confident ones itself: fail → finding, pass → drop. Its unsure residue is not the end of the line - those items are deferred to that check’s post-summary recheck, where the whole-add-on summary re-judges them with the full, line-numbered add-on in view. So the cheap focused pass settles the bulk, and only the genuinely ambiguous cases pay for the whole-add-on look (this is the exact path a permission’s unsure token sites, and a data-exfiltration sink’s unsure verdict, both travel).

The permission recheck is where the summary earns its keep. The unused-permission producer does not just decide whether a permission’s usage tokens are present - it records where each occurs. Each token site (file:line) is handed to the summary as its own item, and the model - seeing the full, line-numbered add-on - judges each: is the permission exercised there? A token-less permission (unlimitedStorage) has no site, so it gets one holistic verdict. resolvePermissionRecheck then aggregates a permission’s sites: any pass drops it (justified), every fail flags it (unused), anything else falls to manual. Because the orchestrator owns the sites, the model only judges given lines - there is nothing to cite and nothing to hallucinate, and a single site’s verdict never decides the whole permission.

With --verbose, the text report surfaces the model’s structured answer: below the “Summary of add-on” prose it prints a Recheck verdicts: list, one bullet per candidate site (the consumer’s title, the [XPI]/[SCA] label, file:line, the subject, and the verdict) with the real source line beneath it. A handed site the model returned no verdict for still appears, as unsure. No model reason is printed - those live in the prose summary above. buildRecheckVerdictReport assembles the rows in runChecks, where each consumer’s corpus is in scope so every source line is read from the right artifact; the renderer only prints them.
